# /retro

## TL;DR

### Intent

Use this to look back over a sprint or week and understand what actually shipped, where effort went, and what patterns are emerging.

### How to use it

Bring it a repo and a time window when you want a higher-signal retrospective than "we were busy."

Use it after a sprint, release cycle, or intense work stretch.

### What is actually useful here

This one is moderately useful if kept simple.

The valuable parts are:

- summarize what shipped
- see where effort clustered
- notice churn hotspots
- look at test versus production work
- ask whether the team is fixing too much, scattering too much, or focusing well

A lot of the original per-person narrative and instrumentation can be cut.

## Core Use

Use this for:

- "What did we ship this week?"
- "Where did our effort actually go?"
- "What patterns should we notice from this sprint?"

## Minimal Flow

1. Choose a time window.
2. Look at commits, major changes, and hotspots.
3. Summarize what shipped.
4. Note patterns in focus, churn, tests, and bug-fixing.
5. End with a few grounded takeaways.

## What It Should Push On

- too much fix-only work
- repeated churn in the same files
- weak test investment
- scattered attention
- strong focus that should be preserved

## Rules

- Do not turn the retro into vanity metrics.
- Do not praise output without looking at quality and pattern.
- Do keep the output short enough to actually read.

## In One Sentence

`/retro` is useful because it turns recent work into a pattern read, not just a list of commits.
