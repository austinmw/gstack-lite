# /codex

## TL;DR

### Intent

Use this when you want a genuinely independent second opinion on a diff, plan, or technical question.

### How to use it

Pick one of three modes:

- review: "give me another review of this diff"
- challenge: "try to break this plan or implementation"
- consult: "answer this hard technical question from scratch"

This is most useful when you want disagreement, not reassurance.

### What is actually useful here

This is only useful if the second voice is actually independent.

The genuinely useful parts are:

- force a fresh review from a different lens
- use adversarial mode instead of polite agreement
- compare overlapping findings and unique findings
- preserve disagreement instead of averaging it away

If it is just the same model paraphrasing itself, most of the value disappears.

## Core Use

Use this for:

- "Give me a second review."
- "Challenge this plan."
- "What would another strong model say about this?"

This is not a default step on every task. It is an outside voice.

## Minimal Flow

1. Choose review, challenge, or consult.
2. Give it the exact artifact or question.
3. Ask for direct output, not flattery.
4. Compare its view with the first-pass view.
5. Keep the new insights, not the duplicated noise.

## What It Should Push On

- blind spots in a plan or diff
- overconfidence from the first review pass
- edge cases a friendly reviewer skipped
- strategic disagreement that deserves to stay visible

## Rules

- Do not treat the second opinion as an oracle.
- Do not ask a fuzzy question and expect a sharp answer.
- Do not hide disagreement just to produce one clean story.
- Do use it when the cost of missing something is high.

## In One Sentence

`/codex` is useful because a real second opinion can surface risks the first model normalized away.
