# /learn

## TL;DR

### Intent

Use this to keep lightweight project memory: recurring pitfalls, preferences, patterns, and lessons that should survive across sessions.

### How to use it

Search it when you suspect the team has seen this issue before.

Prune it when it becomes noisy or stale.

Export it when the best learnings should move into docs.

### What is actually useful here

This is small but real.

The genuinely useful parts are:

- capturing recurring pitfalls so they are not rediscovered every week
- remembering preferences that otherwise get lost between sessions
- pruning stale or contradictory entries
- promoting the best learnings into durable docs

It stops being useful when it turns into a journal or a dump of random trivia.

## Core Use

Use this for:

- "What have we learned on this project?"
- "Did we hit this before?"
- "Clean up or export the project learnings."

This is project memory, not implementation work.

## Minimal Flow

1. Search for the relevant topic or pattern.
2. Read only the few entries that still look useful.
3. Remove or update stale contradictions.
4. Export the best ones into real docs when they matter.

## What It Should Push On

- duplicate discoveries
- stale entries tied to deleted code
- contradictory advice with no timestamp or context
- noisy notes that nobody will search for later

## Rules

- Do not store every observation.
- Do not keep stale learnings just because they exist.
- Do prefer short, searchable, durable statements.

## In One Sentence

`/learn` is useful when it captures recurring lessons and stays aggressively pruned.
