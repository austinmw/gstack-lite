# /plan-ceo-review

## TL;DR

### Intent

Use this when a plan already exists and you want a founder-level challenge on whether it is the right plan, the right scope, and the right version of the idea.

### How to use it

Bring it a design doc, feature plan, or scoped proposal after the initial idea is already somewhat formed.

Use it when you want help deciding whether to expand, hold, or cut the current scope.

### What is actually useful here

The useful part is not the CEO persona. It is the explicit scope posture.

This skill is genuinely useful when it forces the user to pick one of these modes on purpose:

- expand the plan
- selectively expand it
- hold scope and stress-test it
- reduce it to the minimum that matters

That is a real decision most teams blur together.

## Core Use

Use this for:

- "Think bigger."
- "Is this ambitious enough?"
- "Are we solving the right problem?"
- "Is this overbuilt or underbuilt?"

Do not use it as a coding review. It is a plan and scope review.

## Minimal Flow

1. Understand the current plan and the actual outcome it is trying to produce.
2. Challenge the premise: is this solving the real problem or a proxy?
3. Compare at least two plausible paths, not just the draft in front of you.
4. Choose a scope posture explicitly.
5. Rewrite the plan around that choice.
6. Name what is out of scope.

## What It Should Push On

- whether the plan is too timid
- whether it is solving the wrong thing
- whether the scope is inflated without real value
- whether there is a much stronger version nearby
- whether the team is about to spend effort on a second-rate plan

## Rules

- Do not rubber-stamp.
- Do not silently add or cut scope.
- Do not stay neutral forever.
- Do leave behind a sharper recommendation than the user started with.

## In One Sentence

`/plan-ceo-review` is useful because it forces an explicit decision about what version of the plan is actually worth building.
